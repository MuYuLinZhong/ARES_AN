# 12 · 依赖注入：Hilt 模块划分 · Fake/Real 切换策略

> **模块边界**：整个项目的对象创建和依赖装配，是连接所有模块的"胶水层"。  
> **依赖模块**：所有其他模块（提供它们的实现类）  
> **被依赖**：无（最顶层的基础设施）

---

## 1. 为什么用 Hilt

| 不用 DI | 用 Hilt |
| :--- | :--- |
| 每处 `new Repository(new RetrofitService(...))` 手动创建 | 注解声明依赖，框架自动装配 |
| 替换 FakeRepository 需要搜索替换所有地方 | 只改一处 `@Binds`，全局生效 |
| 单元测试无法替换真实依赖 | `@TestInstallIn` 注入假实现 |
| ViewModel 与 Repository 紧耦合 | ViewModel 只依赖接口，不关心实现类 |

---

## 2. Hilt 模块总览

| 模块文件 | 提供内容 | 作用域 |
| :--- | :--- | :--- |
| `NetworkModule` | OkHttpClient, Retrofit, ApiService | `@Singleton` |
| `DatabaseModule` | AppDatabase, 所有 DAO | `@Singleton` |
| `RepositoryModule` | 所有 Repository 接口 → 实现类的绑定 | `@Singleton` |
| `NfcModule` | NfcAdapter | `@Singleton` |
| `PreferencesModule` | DataStore 实例 | `@Singleton` |
| `SecurityModule` | KeystoreManager | `@Singleton` |

## 3. NetworkModule 设计

**文件**：`di/NetworkModule.kt`

提供内容：
- `KeystoreManager`（来自 SecurityModule，注入用于解密 Token）
- `AuthInterceptor`（需要注入 `PreferencesRepository` 和 `KeystoreManager`）
- `OkHttpClient`（注入 `AuthInterceptor` + 证书固定配置）
- `Retrofit`（注入 `OkHttpClient`，配置 BaseUrl 和 JSON 转换器）
- `ApiService`（注入 `Retrofit`，`create(ApiService::class.java)`）

**循环依赖注意**：`AuthInterceptor` 需要 `PreferencesRepository`，`PreferencesRepository` 需要 `DataStore`——这没有循环，只需确保 `DataStore` 通过 `PreferencesModule` 独立提供。

## 6. RepositoryModule 设计（核心：Fake/Real 切换）

**文件**：`di/RepositoryModule.kt`

通过 `@Binds` 将接口绑定到实现类。**切换 Fake/Real 只需修改这一个文件**。

### 6.1 第一阶段（Fake 实现）
```kotlin
@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {

    @Binds @Singleton
    abstract fun bindAuthRepository(impl: FakeAuthRepository): AuthRepository

    @Binds @Singleton
    abstract fun bindDeviceRepository(impl: FakeDeviceRepository): DeviceRepository

    @Binds @Singleton
    abstract fun bindCryptoRepository(impl: FakeCryptoRepository): CryptoRepository

    @Binds @Singleton
    abstract fun bindNfcRepository(impl: FakeNfcRepository): NfcRepository

    @Binds @Singleton
    abstract fun bindPreferencesRepository(impl: DataStorePreferencesRepository): PreferencesRepository
}
```

### 6.2 第二阶段（真实实现，逐步替换）
```kotlin
// 只改 bind 目标，其他所有层零改动
@Binds @Singleton
abstract fun bindAuthRepository(impl: RemoteAuthRepository): AuthRepository  // 改这一行

@Binds @Singleton
abstract fun bindNfcRepository(impl: NfcRepositoryImpl): NfcRepository        // 改这一行
```

## 9. ViewModel 注入

所有 ViewModel 使用 `@HiltViewModel` 注解，通过构造函数注入 UseCase：

```kotlin
@HiltViewModel
class AuthViewModel @Inject constructor(
    private val loginUseCase: LoginUseCase,
    private val logoutUseCase: LogoutUseCase,
    private val updatePasswordUseCase: UpdatePasswordUseCase
) : ViewModel()
```

UseCase 通过 `@Inject constructor` 注入 Repository：

```kotlin
class LoginUseCase @Inject constructor(
    private val authRepository: AuthRepository  // 接口，由 RepositoryModule 绑定实现
)
```

## 11. 测试替换（单元测试）

使用 `@TestInstallIn` 替换 `RepositoryModule`，注入假数据实现：

```kotlin
@TestInstallIn(
    components = [SingletonComponent::class],
    replaces = [RepositoryModule::class]
)
@Module
abstract class TestRepositoryModule {
    @Binds abstract fun bindAuthRepository(impl: FakeAuthRepository): AuthRepository
    // ... 其他绑定
}
```

这样所有集成测试自动使用假数据，不依赖网络，可在 CI 环境运行。
